# Phase 2: ペルソナ駆動調査（STORM方式）

## 目的

調査の死角を構造的に排除するため、仮想専門家ペルソナを生成し、多角的な視点から調査質問を補強する。

## なぜペルソナを使うか

単一の視点で調査すると、調査者のバイアスにより特定の領域が手薄になる。
STORMの研究（Stanford, 28k⭐）により、ペルソナ駆動の調査は網羅性を大幅に向上させることが実証されている。

## 手順

### Step 1: ペルソナ生成

ドメインprompt.mdの `## ペルソナ設定` を参考に、**3〜5人の仮想専門家**を生成する。

#### ペルソナ設計の原則

- **専門領域が重複しない**こと
- **少なくとも1人は批判的・懐疑的な視点**を持つこと（反証担当）
- **少なくとも1人は実務・現場の視点**を持つこと（理論だけでなく実践）
- ドメインprompt.mdにペルソナのヒントがある場合はそれを優先する
- ヒントがない場合は調査テーマから自動生成する

#### ペルソナテンプレート

```
### ペルソナ1: [名前]（[専門領域]）
- 専門: [具体的な専門分野]
- 視点: [この人がこのテーマをどう見るか]
- 重視する情報: [この人が特に注目するデータ・指標]
- 問いかけ: [この人が投げかける質問 2-3個]
```

### Step 2: ペルソナ別質問生成

各ペルソナの視点から、Phase 1のサブ質問を補強する追加質問を生成する。

- 各ペルソナから **2〜3個の追加質問** を出す
- Phase 1のサブ質問と重複するものは除外する
- 「この専門家なら何を聞くか？」を基準にする

### Step 3: 質問の統合・優先度付け

Phase 1のサブ質問 + ペルソナ追加質問を統合し、以下のように整理する：

| # | 質問 | 出典 | 優先度 |
|---|------|------|--------|
| 1 | ○○の現状は？ | Phase 1 | 必須 |
| 2 | △△のリスクは？ | ペルソナ3（懐疑派） | 必須 |
| 3 | □□の実務上の課題は？ | ペルソナ4（実務家） | あれば良い |

### Step 4: 検索クエリへの変換

統合された質問リストを、Phase 3（検索戦略）で使用する検索クエリに変換する。
各質問について **2〜3個のクエリバリエーション** を生成する。

## スケール調整

| 調査規模 | ペルソナ数 | 追加質問数 |
|---------|----------|----------|
| 小規模（単発・狭い） | 3人 | 各2問 |
| 中規模（週次・標準） | 4人 | 各2-3問 |
| 大規模（多国・多軸） | 5人 | 各3問 |

## ペルソナの固定と再利用

- prompt.md に詳細なペルソナ定義がある場合、そのまま使う（自動生成しない）
- Phase 0.5 で自動生成したペルソナは prompt.md に保存され、次回以降も同じペルソナで実行する
- ペルソナを変更するとクエリの軸がずれるため、週次調査では原則ペルソナを固定すること
- ペルソナの変更が必要な場合は、ユーザーに確認を取ってから変更する

## ドメインprompt.mdにペルソナ設定がない場合の自動生成ルール

以下のテンプレートから適切なペルソナを選定する：

1. **業界アナリスト** — 市場動向・数字・トレンドに強い
2. **技術専門家** — 技術的実現性・アーキテクチャに詳しい
3. **規制・法務担当** — 法規制・コンプライアンスの観点
4. **投資家・VC** — 収益性・市場機会・リスクリターンの視点
5. **エンドユーザー/消費者** — 実際に使う側の体験・ニーズ
6. **懐疑派・批評家** — 反証・リスク・失敗シナリオを指摘
7. **現場実務者** — 実装・運用・日常業務の視点
