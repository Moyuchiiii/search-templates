# オーケストレーター（全体フロー制御）

このファイルはサーチの全体フローを定義する。各ドメインのprompt.mdから呼び出され、8フェーズを順次実行する。

## 実行フロー

### Phase 0: 初期化
- `[ドメイン名]/prompt.md` の存在を確認する
- `search/country.md`（必要な場合）を読み込む
- 前回のレポート・track_recordがあれば読み込む（差分追跡用）
- 利用可能なMCPツールを確認する（Exa / Firecrawl / Web検索）
  - 利用不可のMCPがあれば、代替手段を調査結果ファイルの冒頭に記録する
  - Exa/Firecrawlが使えない場合は通常Web検索のみで実行（レポートに「一部ツール未使用」を明記）

```
prompt.md あり？
  ├─ Yes → 読み込んで Phase 1 へ
  └─ No  → Phase 0.5（自動生成）→ ユーザー確認 → Phase 1 へ
```

### Phase 0.5: prompt.md 自動生成（00a-generate-prompt.md）
- **条件**: `[ドメイン名]/prompt.md` が存在しない場合のみ実行
- ユーザーのテーマからペルソナ・カテゴリ・クエリ・スコアリング等を自動生成する
- 生成後、ユーザーの承認を得てから Phase 1 へ進む
- 詳細は `00a-generate-prompt.md` を参照
- **モデル: Sonnet**

```
prompt.md あり？
  ├─ Yes → Phase 1へ
  └─ No → Phase 0.5（自動生成）→ ユーザー確認 → Phase 1へ
```

### Phase 0.5: prompt.md 自動生成（00a-generate-prompt.md）

**実行条件: `[ドメイン名]/prompt.md` が存在しない場合のみ**

- `_template/00a-generate-prompt.md` の手順に従い、ユーザーのテーマから prompt.md を自動生成する
- 生成した内容をユーザーに提示して確認を取る
- **ユーザーの承認が出るまで Phase 1 には進まない**
- 承認後、生成した prompt.md を `[ドメイン名]/prompt.md` に保存してから Phase 1 へ進む

既存の prompt.md がある場合はこのフェーズをスキップする。

### Phase 1: 要件明確化（01-clarify.md）
- 調査目的をサブ質問に分解する
- 調査スコープ・制約・対象期間を定義する
- **モデル: Sonnet**（判断は必要だが高度な推論は不要）

### Phase 2: ペルソナ生成（02-persona.md）
- ドメインに応じた仮想専門家ペルソナを3〜5人生成する
- 各ペルソナの視点からPhase 1のサブ質問を補強・拡張する
- **モデル: Sonnet**

### Phase 3: 検索実行（03-search-strategy.md）
- 検索クエリを生成し実行する
- 多言語検索・適応的クエリ拡張を行う
- 利用可能なスキル（`deep-research` / `exa-search` / `market-research`）があれば活用する
- **モデル: Sonnet**（複数エージェント並列可）

### Phase 4: データ収集（04-data-collection.md）
- 検索結果を構造化して記録する
- 生データファイル（調査結果_YYYY-MM-DD.md）に随時記録する
- **モデル: Sonnet**（大量データの転記はHaiku可）
- ⚠️ **Phase 4完了時点で必ず中間ファイルに保存する。** コンテキスト切れが起きても再開できる状態を維持する

### Phase 5: 検証（05-verification.md）
- 交差検証・コンセンサス判定を行う
- 信頼度タグを付与する
- **モデル: Sonnet**

### Phase 6: 分析（06-analysis.md）
- 情報を圧縮・構造化する
- スコアリング・ランキングを生成する
- **モデル: Sonnet**（複雑な分析判断が必要な場合はOpus）

### Phase 7: レポート生成（07-report.md）
- HTML形式のレポートを出力する
- ドメインprompt.mdのセクション定義に従う
- **モデル: Sonnet**
- ⚠️ HTMLが巨大になりすぎる場合はセクション単位で分割生成し、最後に結合する

### Phase 8: トラックレコード更新（08-track-record.md）
- 前回の予測を検証する
- 今回の予測を記録する
- **モデル: Haiku**（定型処理）

## マルチモデル戦略（STORM方式）

| フェーズ | デフォルトモデル | 切り替え条件 |
|---------|---------------|------------|
| 探索フェーズ（1-4） | Sonnet | 大量の定型転記 → Haiku |
| 分析フェーズ（5-6） | Sonnet | アーキテクチャ判断 → Opus |
| 生成フェーズ（7-8） | Sonnet | 定型更新 → Haiku |

## 並列実行ルール（dispatching-parallel-agents方式）

### 並列化の判断フロー

```
複数のカテゴリ/国がある？
  ├─ Yes → それぞれ独立して調査可能？
  │         ├─ Yes → 並列エージェントをディスパッチ
  │         └─ No（共有状態あり）→ 順次実行
  └─ No → 単一エージェントで実行
```

### エージェントディスパッチのパターン

**パターン1: 国別並列（AI調査等）**
```
Agent("Tier1国の調査", scope="アメリカ,中国", model="sonnet")
Agent("Tier2国の調査", scope="イギリス,韓国,...", model="sonnet")
Agent("Tier3国の調査", scope="日本,UAE,...", model="sonnet")
```

**パターン2: カテゴリ別並列（auto-income等）**
```
Agent("カテゴリ1-3の調査", scope="フリーランス,バウンティ,コンテンツ", model="sonnet")
Agent("カテゴリ4-7の調査", scope="SaaS,トレーディング,データ,ニッチ", model="sonnet")
```

### 各エージェントに渡す情報（必須）

1. **スコープ**: 担当する国/カテゴリの明示的なリスト
2. **テンプレート参照**: 使用する `_template/` ファイルのパス
3. **出力フォーマット**: `04-data-collection.md` のフォーマットに従う旨
4. **クエリリスト**: そのスコープに該当する検索クエリの全リスト
5. **鮮度ルール**: ドメインprompt.mdの鮮度テーブル

⚠️ 「前回と同じ」「適切に判断して」等の曖昧な指示は禁止。全情報を明示的に渡すこと。

### 統合ポイント

- Phase 3-4（検索・収集）は並列化の主要対象
- Phase 5以降はすべてのエージェントの結果が揃ってから実行（部分失敗時は異常系ハンドリングを参照）
- 各エージェントの出力は `調査結果_YYYY-MM-DD.md` の該当セクションに統合する

## フェーズ完了マーカー

各フェーズが完了したら、調査結果MDの該当セクション末尾に完了マーカーを書き込む。

```
<!-- Phase N 完了: YYYY-MM-DD HH:MM -->
```

- Phase 3-4の並列実行時は、各エージェントが**自分の担当セクションの末尾**にマーカーを書く
- マーカーがないフェーズは「未完了」として扱い、Phase 5以降の実行前に確認する

## 異常系ハンドリング

### 並列エージェントの部分失敗

**タイムアウト（10分応答なし）:**
- 応答のないエージェントを切り上げ、利用可能なデータのみで次フェーズに進む
- レポートの「データ品質」セクションに「一部データ未取得: [担当スコープ]」を明記する
- 部分データでもPhase 5以降は実行する

**エージェントエラー:**
- エラー内容を調査結果ファイルに記録する
- 単純な再試行で解決しない場合は、残りのエージェント結果のみで続行する

### 検索ゼロヒット時

- クエリ変更は**最大3回まで**。語順変更・表記ゆれ・英語クエリ等を試みる
- 3回変えてもゼロヒットの場合は「情報なし（クエリ枯渇）」として記録し、次のクエリへ進む
- カテゴリ全体でゼロヒットの場合、そのカテゴリを「調査不可」としてレポートに記載する

### MCP利用不可時

- Phase 0でMCPの利用可否を確認済みの場合は、記録した代替手段で実行する
- Exa/Firecrawl両方が使えない場合は、通常Web検索のみで実行する
- レポートの冒頭に「検索ツール制限あり: Exa/Firecrawl 未使用のため網羅性が低下している可能性あり」を明記する

### コンテキストウィンドウ対策

- Phase 4完了時点で必ず中間ファイル（`調査結果_YYYY-MM-DD.md`）に全データを保存する
- コンテキスト切れが起きた場合は、中間ファイルを読み込んでPhase 5から再開する
- Phase 7でHTMLが巨大になりすぎる場合は、セクション単位で分割生成してから結合する

## 出力ファイル

| ファイル | フェーズ | 説明 |
|---------|---------|------|
| `調査結果_YYYY-MM-DD.md` | Phase 3-5 | 生データ（検索ログ・収集情報・検証結果） |
| `レポート_YYYY-MM-DD.html` | Phase 7 | 最終レポート（HTML単一ファイル） |
| `track_record.md` | Phase 8 | 予測トラックレコード（追記） |

## バッチ実行モード（推奨）

> 単一エージェントで全8フェーズを通して実行する「直列モード」も引き続き有効。
> ただし **バッチモードを推奨する**。コンテキストが2分割されるため、特にPhase 3-5のような大量データ収集フェーズでのコンテキスト圧迫を防げる。

### 概要

8フェーズを「調査バッチ」と「レポートバッチ」の2つに分割し、別々のエージェントコンテキストで実行する。
バッチ間のデータ受け渡しは `調査結果_YYYY-MM-DD.md`（中間ファイル）で行う。

```
[バッチ1: 調査バッチ]              [バッチ2: レポートバッチ]
 調査エージェント                    レポートエージェント
 Phase 0〜5 を実行          →       Phase 6〜8 を実行
 読み込み:                          読み込み:
   00-orchestrator.md                 00-orchestrator.md
   01-clarify.md                      06-analysis.md
   02-persona.md                      07-report.md
   03-search-strategy.md              07a-report-css.md  ← Phase 7開始時
   04-data-collection.md              07b-report-html.md ← Phase 7開始時
   05-verification.md                 08-track-record.md
   ドメインprompt.md                  ドメインprompt.md
 出力:                               入力:
   調査結果_YYYY-MM-DD.md      →       調査結果_YYYY-MM-DD.md
                                      （バッチ1の出力を読む）
```

### バッチ1: 調査バッチ（Phase 0〜5）

- **担当**: 調査エージェント（Sonnet）
- **読み込むテンプレート**: `00-orchestrator.md`, `01-clarify.md` 〜 `05-verification.md`, ドメインprompt.md
- **読み込まないもの**: `06-analysis.md`, `07-report.md`, `07a-report-css.md`, `07b-report-html.md`, `08-track-record.md`
- **出力**: `[ドメイン名]/YYYY-MM-DD/調査結果_YYYY-MM-DD.md`（フェーズ完了マーカー付き）
- **完了条件**: Phase 5 の完了マーカーが中間ファイルに書き込まれていること

### バッチ2: レポートバッチ（Phase 6〜8）

- **担当**: レポートエージェント（Sonnet）。バッチ1とは別コンテキストで起動する
- **読み込むテンプレート**: `00-orchestrator.md`, `06-analysis.md`, `07-report.md`, `08-track-record.md`, ドメインprompt.md
- **読み込まないもの**: `01-clarify.md` 〜 `05-verification.md`（バッチ1の調査テンプレートは不要）
- **07a / 07b の読み込みタイミング**: Phase 7 の実行開始時のみ読み込む（Phase 6・8の実行中は不要）
- **入力**: `[ドメイン名]/YYYY-MM-DD/調査結果_YYYY-MM-DD.md`
- **出力**: `[ドメイン名]/YYYY-MM-DD/レポート_YYYY-MM-DD.html`, `[ドメイン名]/track_record.md`（追記）

### バッチ間の受け渡し手順

1. バッチ1完了後、`調査結果_YYYY-MM-DD.md` が保存されていることを確認する
2. バッチ2のエージェントを起動し、以下を明示的に渡す:
   - 中間ファイルのフルパス（例: `search/AI/2026-03-25/調査結果_2026-03-25.md`）
   - ドメインprompt.mdのパス
   - 実行するフェーズ（Phase 6〜8）
   - 使用するテンプレートのパス一覧
3. バッチ2エージェントは中間ファイルを読み込み、Phase 6 から実行を開始する

### バッチモードの利点

| 比較軸 | 直列モード | バッチモード |
|--------|-----------|------------|
| コンテキスト効率 | 全テンプレートを最初から保持 | 各バッチで必要なテンプレートのみ保持 |
| CSS/HTMLの読み込み | Phase 1から保持（905+348行） | Phase 7開始時のみ読み込み |
| 調査テンプレートの保持 | Phase 7まで保持し続ける | バッチ2では不要・読み込まない |
| 再開しやすさ | コンテキスト切れ時に復元が難しい | 中間ファイルから容易に再開可能 |
| 並列化 | 基本的に順次 | バッチ1並列 + バッチ2は1エージェント |

## ドメインprompt.mdの必須セクション

各ドメインのprompt.mdには以下のセクションが必須：

```
## 基本情報
- 調査名、頻度、保存先

## 事前読み込み
- 調査開始前に読むべきファイルのリスト

## ペルソナ設定
- 仮想専門家のヒント（02-persona.mdが自動生成する際の方向性）

## 調査カテゴリ
- サブ質問分解の軸

## 検索クエリ
- 固定クエリリスト（毎回実行するもの）

## スコアリング設定（任意）
- 評価軸と重み

## レポートセクション
- HTML レポートのセクション構成

## 文脈連携（任意）
- 他の調査レポートとの連携ルール
```
