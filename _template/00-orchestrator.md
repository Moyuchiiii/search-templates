# オーケストレーター（全体フロー制御）

このファイルはサーチの全体フローを定義する。各ドメインのprompt.mdから呼び出され、8フェーズを順次実行する。

## 実行フロー

### Phase 0: 初期化
- ドメインのprompt.mdを読み込む
- `search/country.md`（必要な場合）を読み込む
- 前回のレポート・track_recordがあれば読み込む（差分追跡用）
- 事前読み込みファイル（prompt.mdの`## 事前読み込み`セクションで指定）を読み込む

### Phase 1: 要件明確化（01-clarify.md）
- 調査目的をサブ質問に分解する
- 調査スコープ・制約・対象期間を定義する
- **モデル: Sonnet**（判断は必要だが高度な推論は不要）

### Phase 2: ペルソナ生成（02-persona.md）
- ドメインに応じた仮想専門家ペルソナを3〜5人生成する
- 各ペルソナの視点からPhase 1のサブ質問を補強・拡張する
- **モデル: Sonnet**

### Phase 3: 検索実行（03-search-strategy.md）
- 検索クエリを生成し実行する
- 多言語検索・適応的クエリ拡張を行う
- **モデル: Sonnet**（複数エージェント並列可）

### Phase 4: データ収集（04-data-collection.md）
- 検索結果を構造化して記録する
- 生データファイル（調査結果_YYYY-MM-DD.md）に随時記録する
- **モデル: Sonnet**（大量データの転記はHaiku可）

### Phase 5: 検証（05-verification.md）
- 交差検証・コンセンサス判定を行う
- 信頼度タグを付与する
- **モデル: Sonnet**

### Phase 6: 分析（06-analysis.md）
- 情報を圧縮・構造化する
- スコアリング・ランキングを生成する
- **モデル: Sonnet**（複雑な分析判断が必要な場合はOpus）

### Phase 7: レポート生成（07-report.md）
- HTML形式のレポートを出力する
- ドメインprompt.mdのセクション定義に従う
- **モデル: Sonnet**

### Phase 8: トラックレコード更新（08-track-record.md）
- 前回の予測を検証する
- 今回の予測を記録する
- **モデル: Haiku**（定型処理）

## マルチモデル戦略（STORM方式）

| フェーズ | デフォルトモデル | 切り替え条件 |
|---------|---------------|------------|
| 探索フェーズ（1-4） | Sonnet | 大量の定型転記 → Haiku |
| 分析フェーズ（5-6） | Sonnet | アーキテクチャ判断 → Opus |
| 生成フェーズ（7-8） | Sonnet | 定型更新 → Haiku |

## 並列実行ルール

- Phase 3（検索）は国別・カテゴリ別にエージェントを並列起動できる
- Phase 4（データ収集）も情報ソースごとに並列可
- Phase 5以降は前フェーズの完了を待つ（順次実行）

## 出力ファイル

| ファイル | フェーズ | 説明 |
|---------|---------|------|
| `調査結果_YYYY-MM-DD.md` | Phase 3-5 | 生データ（検索ログ・収集情報・検証結果） |
| `レポート_YYYY-MM-DD.html` | Phase 7 | 最終レポート（HTML単一ファイル） |
| `track_record.md` | Phase 8 | 予測トラックレコード（追記） |

## ドメインprompt.mdの必須セクション

各ドメインのprompt.mdには以下のセクションが必須：

```
## 基本情報
- 調査名、頻度、保存先

## 事前読み込み
- 調査開始前に読むべきファイルのリスト

## ペルソナ設定
- 仮想専門家のヒント（02-persona.mdが自動生成する際の方向性）

## 調査カテゴリ
- サブ質問分解の軸

## 検索クエリ
- 固定クエリリスト（毎回実行するもの）

## スコアリング設定（任意）
- 評価軸と重み

## レポートセクション
- HTML レポートのセクション構成

## 文脈連携（任意）
- 他の調査レポートとの連携ルール
```
