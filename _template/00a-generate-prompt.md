# Phase 0.5: prompt.md 自動生成

prompt.md が存在しない新規ドメインでのみ実行するフェーズ。
ユーザーが指定したテーマから、即座に本番レベルの調査設定を自動生成する。

## 前提条件

- このフェーズは `[ドメイン名]/prompt.md` が存在しない場合にのみ実行する
- 既存の prompt.md がある場合はこのフェーズをスキップして Phase 1 へ進む
- **モデル: Sonnet**

## 入力

ユーザーから以下の情報を受け取る:

1. **テーマ**（必須）: 調査したい領域。例: 「AI副業」「仮想通貨投資」「Webマーケティング」「ノーコードツール」
2. **ドメイン名**（必須）: フォルダ名に使う短い英語名。例: `ai-side-hustle`, `crypto-invest`
3. **調査目的**（任意）: 「投資判断」「新規ビジネス探索」「学習」等
4. **特に気にする観点**（任意）: 「コスト重視」「自動化度重視」「リスク最小化」等

テーマとドメイン名だけでも生成を開始できる。他の情報は生成時に自動的に推論する。

## 生成する内容

### 1. 基本情報

```
## 基本情報

- 調査名: [テーマ]に関する調査
- 頻度: 随時
- 保存先: search/[ドメイン名]/YYYY-MM-DD/
- 調査対象期間: 直近1〜3ヶ月を基本
```

### 2. ペルソナ設定（4〜5人）

必ず以下の条件を満たすペルソナセットを作る:

- **実践者**: 実際にそのテーマで活動している人物（具体的な年収・実績を設定）
- **専門家**: 業界・技術・法律等の専門知識を持つ人物
- **ビジネス視点**: コスト・収益・スケーラビリティを評価できる人物
- **リスク評価者**: 法規制・プラットフォームリスク・持続性に詳しい人物
- **懐疑派**（必須）: 「それって本当に機能するの？」と突っ込む批判的思考の持ち主。楽観的な見通しに異議を唱える役割

ペルソナ例（投資系テーマの場合）:
```
- 個人投資家（7年以上の運用実績、年利15%超の実績あり）
- フィンテックエンジニア（自動売買システムの設計・運用経験）
- 金融規制アドバイザー（金商法・暗号資産交換業に詳しい）
- リスク管理の専門家（ドローダウン・破産確率の定量評価ができる）
- 懐疑派の元デイトレーダー（多くの「自動化手法」が実際には機能しないことを知っている）
```

### 3. 調査カテゴリ（5〜9個）

MECE原則（漏れなく・重複なく）で分類する。

生成ルール:
- テーマを構成する主要な側面をすべてカバーする
- 各カテゴリに「調査項目」を5〜8個列挙する
- 必ずリスク・法規制・失敗事例を扱うカテゴリを1つ含める
- テーマによっては国際比較・日本特有の機会・競合状況を独立カテゴリにする

例（Webマーケティングテーマの場合）:
```
1. SEO・オーガニック集客
2. 有料広告（Google/Meta/TikTok）
3. SNSマーケティング・インフルエンサー
4. コンテンツマーケティング・メール
5. データ分析・CVR最適化
6. ツール・自動化・AI活用
7. 法規制・景品表示法・個人情報
8. コスト構造・ROI分析
9. 新興手法・実験的アプローチ
```

### 4. 検索クエリ（最低30本）

品質の高いクエリセットを生成する。内訳:

- **英語クエリ**: 最低20本
- **日本語クエリ**: 最低10本

クエリ生成ルール:
- **完全一致フレーズ**: `"exact phrase 2026"` 形式で主要キーワードを囲む
- **サイト指定**: `site:reddit.com`, `site:news.ycombinator.com`, `site:twitter.com` を含むクエリを各3本以上
- **実績・証拠を求めるクエリ**: `proof`, `case study`, `income report`, `成功事例`, `実績` を含む
- **リスク・失敗を求めるクエリ**: `risks`, `failure`, `warning`, `scam`, `失敗事例`, `リスク` を含む
- **最新情報**: `2026` または `2025` を含むクエリを半数以上
- **比較クエリ**: `vs`, `comparison`, `best`, `alternatives` を含む
- **日本市場特有**: 日本のプラットフォーム・法規制・市場環境に関するクエリ

クエリは各カテゴリに対応するサブセクションで整理する。

### 5. スコアリング設定

8軸で評価。テーマに応じて重みを調整する。

**標準テンプレート:**

| 評価軸 | 重み | 重みの調整指針 |
|--------|------|--------------|
| 実現性 | 25% | 固定（実装可能かは常に最重要） |
| 収益性 | 20% | 投資系は25%に上げ、学習目的は10%に下げる |
| リスク（逆） | 15% | 投資系・法規制リスクが高い領域は20%に上げる |
| 持続性 | 15% | プラットフォーム依存度が高い領域は20%に上げる |
| 競合度（逆） | 10% | レッドオーシャン調査が目的の場合は15%に上げる |
| 初期コスト（逆） | 7% | 資金調達困難な場合は12%に上げる |
| スケーラビリティ | 5% | ビジネス拡大が目的の場合は10%に上げる |
| 実例の有無 | 3% | 固定（証拠ベース評価として維持） |

**重みの自動調整ルール:**
- テーマに「投資」「トレーディング」が含まれる → リスク(逆)を20%に上げ、収益性を25%に
- テーマに「副業」「フリーランス」が含まれる → 実現性を30%に上げ、収益性を25%に
- テーマに「SaaS」「プロダクト」が含まれる → スケーラビリティを15%に上げ、初期コストを10%に

**実現性ランク定義（共通）:**

| ランク | 総合スコア | 定義 |
|--------|-----------|------|
| S | 85〜100 | 今すぐ実装可能。類似の仕組みが既に動いている |
| A | 70〜84 | 1-2週間で実装可能。技術的なハードルは低い |
| B | 55〜69 | 1ヶ月程度で実装可能。一部検証が必要 |
| C | 40〜54 | 実装可能だが課題多い。規約・法的リスクあり |
| D | 20〜39 | 理論上は可能だが現実的でない |

### 6. 鮮度ルール

標準テンプレートをベースに、テーマの情報変化速度に応じて調整する。

**標準（多くのテーマに適用）:**

| 鮮度 | 重み倍率 | 扱い |
|------|---------|------|
| 1ヶ月以内 | ×1.5 | 最優先 |
| 3ヶ月以内 | ×1.3 | 主要情報 |
| 6ヶ月以内 | ×1.0 | 標準 |
| 1年以内 | ×0.7 | 背景情報（日付明記） |
| 1年超 | ×0.3 | 原則使用しない |

**調整基準:**
- 変化が速いテーマ（仮想通貨、AI、SNS等）: 1ヶ月以内を×2.0に強化
- 変化が遅いテーマ（不動産、製造業等）: 1年以内も×0.9程度で使用可

### 7. レポートセクション構成（7〜12セクション）

テーマに合わせてセクション名・内容を決定する。

**必須セクション（テーマに関わらず含める）:**
1. エグゼクティブサマリー（調査ハイライト、即実行すべき TOP3、前回との差分）
2. ランキング / 比較テーブル（スコア付き、ソート可能）
3. カテゴリ別詳細分析
4. リスクマトリクス（法規制・失敗リスク・実現性リスク）
5. アクションプラン（今すぐ / 1ヶ月 / 3ヶ月に分けた具体的ステップ）
6. 今後の注目領域（次の調査サイクルで追うべき動き）
7. おすすめリソース TOP5（有益だったソース、継続ウォッチ推奨）

**テーマに応じて追加するセクション（例）:**
- 投資系: 実績トラックレコード、ポートフォリオ戦略、国別比較
- 副業系: 実装ロードマップ、収益シミュレーション、組み合わせ戦略
- ビジネス系: 市場規模分析、競合マッピング、顧客セグメント

**HTMLデザイン要件（共通）:**
- `_template/07-report.md` のデザイン仕様に従う
- 単一ファイル（CSS・JSはすべてインライン埋め込み）、UTF-8
- 白ベースのモダンデザイン（デフォルトテーマ: light）
- レスポンシブ対応

## 実行手順

### Step 1: テーマ分析

テーマを受け取ったら、まず以下を内部で分析する:

- テーマの主要な側面（何を調査すべきか）
- 情報の変化速度（日次/週次/月次）
- 主なステークホルダー（誰の視点が必要か）
- 代表的なプラットフォーム・サービス（何を調べるか）
- リスクの種類（規制/技術/市場/詐欺）
- 日本市場の特殊性（グローバルと異なる点）

### Step 2: prompt.md を生成

上記の分析をもとに、上で定義した全セクションを含む prompt.md を生成する。

生成時の注意:
- クエリは「思いつき」ではなく、調査で実際に使えるレベルの具体性にする
- スコアリングの重みはテーマの特性に合わせて明示的に調整する
- ペルソナは架空でも具体的に書く（「Aさん（38歳）は月収50万円のフリーランサーで...」等）

### Step 3: ファイルに保存

生成した内容を `[ドメイン名]/prompt.md` に書き込む。

### Step 4: ユーザーに提示して確認

生成した prompt.md の内容をユーザーに提示して、以下を確認する:

```
prompt.md を生成しました。以下の内容で調査を進めてよいですか？

【ペルソナ】
（生成したペルソナ一覧）

【調査カテゴリ】
（生成したカテゴリ一覧）

【スコアリング重み調整】
（標準からの変更点があれば明示）

【クエリ総数】 英語XX本 / 日本語XX本

修正点があれば教えてください。OKであれば Phase 1 に進みます。
```

**ユーザーの承認を得るまで Phase 1 には進まない。**

修正が必要な場合は内容を修正して再提示する。「OK」「いいよ」「進めて」等の承認が出たら Phase 1 へ。

## 品質チェックリスト

生成後、保存前に以下を自己チェックする:

- [ ] ペルソナは5人いるか（うち少なくとも1人は懐疑派）
- [ ] カテゴリは5個以上あり、MECE的に整理されているか
- [ ] 英語クエリが20本以上あるか
- [ ] 日本語クエリが10本以上あるか
- [ ] `site:reddit.com`, `site:news.ycombinator.com` を含むクエリがあるか
- [ ] `proof`, `case study`, `実績` 等の証拠を求めるクエリがあるか
- [ ] リスク・失敗事例を調査するクエリがあるか
- [ ] スコアリングの重みの合計が100%になっているか
- [ ] レポートセクションが7個以上あるか
- [ ] 必須セクション7つがすべて含まれているか
