# Phase 6: 分析・スコアリング

## 目的

Phase 4-5で検証済みの情報を構造化し、スコアリング・ランキング・トレンド分析を行う。

## 手順

### Step 1: 情報圧縮（open_deep_research方式）

大量の生データをレポートに変換する前に、情報を圧縮・構造化する。

#### 圧縮ルール
- 同じトピックについて複数ソースがある場合、最も信頼性の高いソースを主軸にする
- 重複情報は統合し、追加ソースは補強として引用する
- 数値データは最新の出典を優先する
- 圧縮後も**ソースURLは必ず保持する**（トレーサビリティ確保）

#### 圧縮フォーマット

```
### [トピック名]

**要点**: [1-2文で核心を記述]
**詳細**: [補足情報。3-5文]
**数値**: [具体的な数字があれば]
**ソース**: [URL1], [URL2]
**信頼度**: [検証済/未検証] / コンセンサス[高/中/低]
**鮮度**: [情報の日付]
```

### Step 2: スコアリング（ドメインprompt.md連動）

ドメインprompt.mdに `## スコアリング設定` がある場合、その軸と重みに従ってスコアリングする。

#### 汎用スコアリングフレームワーク

スコアリング設定がない場合でも、以下の汎用フレームワークを適用できる：

```
総合スコア = Σ (各軸のスコア × 重み)

各軸のスコア: 1〜10の整数
重み: 0〜1（合計1.0）
信頼度補正: スコアに検証状態の係数を掛ける
  - 検証済み: ×1.0
  - 未検証: ×0.7
  - ソース不足: ×0.5
```

#### 各評価軸のスコア定義（1〜5点）

スコアは 1〜5 の整数で付ける。各軸の定義は以下のとおり。

**実現性**（技術的・実務的に今すぐ取り組めるか）

| 点数 | 定義 |
|------|------|
| 5 | 今すぐ実装可能。類似の仕組みが既に稼働中 |
| 4 | 1-2週間で実装可能。小さな技術検証だけで済む |
| 3 | 1ヶ月程度。一部の技術検証・法的確認が必要 |
| 2 | 実装可能だが課題多い。規約・法的リスクの対処が先 |
| 1 | 理論上は可能だが現実的でない |

**収益性**（利益を生む規模・確実性）

| 点数 | 定義 |
|------|------|
| 5 | 実績ある収益モデル。月10万円超の収益が複数事例で確認済み |
| 4 | 概算で月5万円以上が見込める。収益構造が明確 |
| 3 | 収益化できる見込みはあるが、金額・時期は不確実 |
| 2 | 収益化まで長期間かかる、または単価が極めて低い |
| 1 | 収益化の見通しが立たない、または赤字リスクが高い |

**リスク（逆）**（低リスクほど高得点）

| 点数 | 定義 |
|------|------|
| 5 | リスクがほぼない。規約・法的問題なし、プラットフォーム依存なし |
| 4 | 軽微なリスクのみ。対処法が明確で影響範囲が小さい |
| 3 | 中程度のリスク。注意が必要だが回避・軽減策がある |
| 2 | 高リスク。規約違反・法的グレーゾーン・垢BAN等の可能性あり |
| 1 | 重大リスク。詐欺・違法・損失が現実的に起こりうる |

**持続性**（長期的に継続できるか）

| 点数 | 定義 |
|------|------|
| 5 | 参入障壁があり長期安定。アルゴリズム変動・競合に強い構造 |
| 4 | 1〜2年は安定が見込める。多少の参入障壁あり |
| 3 | 半年〜1年は機能する見込み。定期的な見直しが必要 |
| 2 | 短命の可能性あり。プラットフォーム変更・競合増加で消える可能性 |
| 1 | トレンド依存・一時的な抜け穴に近い。長期継続は困難 |

**競合度（逆）**（低競合ほど高得点）

| 点数 | 定義 |
|------|------|
| 5 | ほぼ競合なし。ブルーオーシャン状態 |
| 4 | 競合は少ない。差別化で十分勝負できる |
| 3 | 競合は中程度。独自強みがあれば勝てる |
| 2 | 競合が多い。差別化が難しく価格競争になりやすい |
| 1 | 極めてレッドオーシャン。新規参入の余地がほぼない |

**初期コスト（逆）**（低コストほど高得点）

| 点数 | 定義 |
|------|------|
| 5 | 初期コストほぼゼロ。無料ツール・既存スキルで開始可能 |
| 4 | 初期投資1万円未満。サブスク費用程度 |
| 3 | 初期投資1〜10万円程度。回収見込みがある |
| 2 | 初期投資10〜50万円。リスクを取る必要がある |
| 1 | 50万円超の初期投資が必要。または回収不透明 |

**スケーラビリティ**（仕組み化・拡大できるか）

| 点数 | 定義 |
|------|------|
| 5 | 完全自動化可能。収益が人手をほぼ必要とせずに拡大できる |
| 4 | 部分的に自動化できる。少人数で大きな規模を扱える |
| 3 | 一定の手作業が必要だが、外注・仕組み化で拡大できる |
| 2 | 拡大には人員・コストが比例して増える |
| 1 | 完全に工数依存。スケールアップの方法がない |

**実例の有無**（実際に成功している事例があるか）

| 点数 | 定義 |
|------|------|
| 5 | 複数の具体的な成功事例あり。数字・方法論が公開されている |
| 4 | 成功事例あり。一部のみ具体的で信頼性は高い |
| 3 | 成功事例は存在するが少ない、または情報が曖昧 |
| 2 | 成功事例が見当たらない。自称・未検証のもののみ |
| 1 | 成功事例なし。理論や推測のみ |

| 順位 | 対象 | 軸1 | 軸2 | ... | 総合スコア | 信頼度 |

### Step 3: トレンド分析

前回レポートが存在する場合、以下の差分分析を行う：

- **新規**: 今回初めて登場した情報・トレンド
- **変化**: 前回から状態が変わった情報（`[変化]`タグ）
- **継続**: 前回から変化なしの情報
- **消失**: 前回はあったが今回確認できなくなった情報

### Step 4: So What? テスト

各分析結果に対して「で、結局どうなの？」テストを行う：

- この情報は読者にとってなぜ重要か？
- 読者は何をすべきか？（アクション）
- 何も思いつかない場合、その情報はレポートから除外を検討する

### Step 5: 市場分析フレームワーク（market-research方式）

ドメインprompt.mdが市場調査・競合分析を含む場合、以下のフレームワークを適用する。

#### 市場規模の推定（必須: トップダウン + ボトムアップ）

どちらか一方だけでは不十分。両方で検証する:

- **トップダウン**: 業界レポート・統計データから全体市場→対象セグメントに絞り込む
  - 例: 「AI市場$500B → そのうちAI SaaS市場$80B → 中小企業向け$15B」
- **ボトムアップ**: 顧客数×単価×頻度で積み上げる
  - 例: 「想定顧客1,000社×月額$200×12ヶ月 = $2.4M/年」

#### 競合分析テンプレート

| 項目 | 自社/対象 | 競合A | 競合B |
|------|---------|-------|-------|
| 価格帯 | | | |
| 主要機能 | | | |
| ターゲット | | | |
| 強み | | | |
| 弱み | | | |
| 差別化ポイント | | | |

#### 意思決定マトリクス

スコアリングだけでなく、最終的な推奨には以下の視点を加える:

- **リスク調整後リターン**: 高スコアでもリスクが高ければ推奨度を下げる
- **実行可能性**: 技術・リソース・時間の制約を考慮
- **タイミング**: 今やるべきか、待つべきか（市場の成熟度）

## 信頼度スコア（情報充実度）

各カテゴリ/国の情報充実度を5段階で評価する：

| スコア | 基準 |
|--------|------|
| ★★★★★ | 複数の独立ソースで十分な情報。検証済み多数 |
| ★★★★☆ | 主要な動向は把握できている。一部未検証あり |
| ★★★☆☆ | 基本情報は取得できたが網羅性に不安 |
| ★★☆☆☆ | 情報が限定的。記載内容の精度に注意 |
| ★☆☆☆☆ | ほぼ情報なし。「⚠️ 情報不足」と明記 |
